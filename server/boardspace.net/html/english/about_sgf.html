All the game records saved at Boardspace are in SGF format, a used game file format that is mainly used for the game Go.  The skeleton of that format, and basic syntax conventions, are fixed, and expandable enough  to accommadate a wide variety of games.   This file documents the overall adaptions you'll find in Boardspace' sgf files.

Here's an abbreviation of a typical sgf format file at Boardspace.
 (;

CM[0,1]

DT[Thu May 18 17:03:46 PDT 2023]
GN[MA-AntiSkub-ddyer-2023-05-19-0003]

GM[Majorities]
The
name of the game, generally the geneneric name for the base game.
 SGF format expects the GM propertu to be an integer "game
number", which was used for the first few games at boardspace, but it
rapidly became too cumbersome to beg the sgf comittee to assign new
integers.
VV[1]
This
is the format of the SGF file itself, and should always have the value
1.  If there ever were to be a drastic incompatible change, this
would become 2
CM[0,1]
This
is the mapping of colors from their "normal" values,whatever that means
for the game.  This property is not present in older game records.
TC[None]
Indicates the type of time controls for this game.  The default is None, and this property may not always be present.
SU[Majorities-5 2 18340105 100]
Indicates
the details of fixed setup for this game.  This field normally
identifies the exact variation, board size and so on, but the details
will vary depending on the game and the vintage of the game record.
 The "best practice" for new games is to include "variation"
"number of players" "random number seed" and "revision number".
 So for example, this record is for
  1. Majorities played on a 5x board
  2. 2 players
  3. 18340105
as a random number seed.  This seed affects things such as deck
shuffles and dice rolls, which will depend on the exact sequence of
random numbers generated from this seed.
  4. 100 is the revision/bug fix level
It's
standard to include the number of players in the SU property, even for
games which always have the same number of players, and to include the
random number seed even for games which have no randomness.
The
revision number is a clue to viewers to what bug fixes or other changes
may be included in this game record. and will affect the interpretation
of the game record.  In effect, this value allows games to be
replayed in "backward compatability" mode with the rules and features
as they were at the time.  For simple games, it will
probably never change, but complex games developed over a long period
of time, it might change frequently.   For example, Viticulture
started at 100 but is currently at revision 155.  The details are
highly game dependent - consult the developer and the source code!

Note that this property is likely to change over time and acquire more properties if you look back at earlier game records. 
DT[Thu May 18 17:03:46 PDT 2023]
The
date the game was started, in whatever format the local device reported
it.  There is a lot of parochial variation, and some records
include unusual character sets.
GN[MA-AntiSkub-ddyer-2023-05-19-0003]
This
is the unique game name assigned when the game was played.  Some
old game records do not contain this as an internal property, or include a less specfic, non-unique name.
RE[Game won by ddyer]
The result of the game,  not always exactly the same format over time, and not present in some older game records/
P0[id "AntiSkub"]
P1[id "ddyer"]
The name and possibly other properties of the players
; 
	The first semicolon sepatates the general game properties from the actual record of the game. 
  • P0[0 Start P0]
    ; P0[1 Pick W]TM[11896]
    ; P0[2 Dropb W A 5]TM[13462]
    ; P0[3 Done ]TM[14526]
    ; P1[4 Dropb B E 5]TM[2325]
    ; P1[5 Dropb B H 2]TM[4851]
    ; P1[6 Done ]TM[6703]
The
actual record of the game.  This part of potentially different for
every kind of game, but boardspace generally used P0 P1 P2 and so on to
identify the players.
The details inside the [] for each move are
highly variable per game.  Additional properties for each move are
optional.   Newer game recrods will contain TM
properties, which is cumulative player time in milliseconds.   Game records may also contain C[] properties which are comments per move, and AN[] annotation
properties.

Other wrapper details:
  • There
may be more than one game per file.  The multiple games could be a
deliberate collection, or it part of an automatically saved game, it's
an accident from some past debugging, and the second copy of the game
should be ignored.
  • Zip files are packaged automaically from recently played games.
  • Games may contain variations in standard sgf  format.
