Evolution of a Layout Manager


Executive summary:

Over time, as the game space evolved and the requirements for game screen layouts became more complex, the basic construction of  layouts has changed several times.

The Problem Space:

It's alll about rectangles.  The game window is a rectangle, the board is a rectangle, most of the other items in a game window are naturally rectangles.  From there, the problem gets a lot more complicated.   Some rectangles have to be a certain size, or should be placed near other items.  Some are optional.  Some are fixed size, or have a minimum size, or can reasonably expand to fill the space available.

 

In the beginning:

In the beginning I had a basic template for all (about 3 at the time) games.   The window was rectangular, with a chat window on top,
a board in the lower left, game specific information to the right,  and maybe some additional information below the board.


To scale and locate all the elements, I used the board grid as a sort of unit for allocation.  And lay out the other items by size and
location based on the cell size of the board.   Plateau seen at the left, would need a 4x4 grid for the board, or 5x5 with borders all
the way around.   The chat should be about 10 cells (twice as wide as the board) and the game log 3 cells wide.   Player stuff about
5 cells wide and 3 cells high (per player) ... and so on.  Add up all the vertical and horizontal cell counts, do some simple arithmetic,
and come up with a pixel count for  a cell.  Then lay out the rectangles accordingly.

This worked pretty well at first.  The scheme produced reasably full screens with relatively little wasted space.


Wide and Tall boards (3 basic layouts instead of 1)


Next, it became obvious that sometimes people preferred (or were constrained) to use very wide or very tall window shapes.  The basic plan, above, didn't produce good results.  The basic board plan evolved to include "wide mode" and "tall mode" windows.   Wide mode windows (seen below) had the board occupying essentially all the vertical space, with player information adjacent, and the chat window filling space at the right.   Tall mode windows (seen below) had the board occupy substantially all horizontal space, with the chat above and the player information below the board.   Initially, the choice of wide/normal/tall was ad-hoc based on the aspect ratio of the window, but the overall method was the same.

     




No Chat boards (6 basic designs per game)

Some very small screens just don't have room for a chat area, so it's moved to a separate window.  In a lot of games,
particularly games played against robots, the chat is pretty surpufluous - no one chats with a robot.  So removing the
chat can free up a lot of real estate, but the business of deciding where all the boxes are going to go is becoming complex

The algorithm is the same - place the boxes relative to one another, add up the number of cells needed, do arithmetic to
determine the cell size.
   



Multiple Players

Accomodating games with up to 6 players introduced complication.  A game with 6 players  has very different space requirements than the same game with only 2 players; and depending on the amount of per-player information, the best layout can vary a lot.  Think: 1-3 columns right of the board.  1-3 columns or below the board.
